## #9 verdict validator — works as a live nudge, NOT a gate substitute

**It fired and its judgments were substantive.** Run #2: 6 `verdict_validation`
calls, 3 live FLAGs, each charter-accurate:
- H1 FALSIFIED → 🚩 **§4 goalpost-move**
- H1 INCONCLUSIVE (retry) → 🚩 **§3 (an adequate test that contradicts must be FALSIFIED)**
- H2 SUPPORTED → 🚩 **§2–§3 (premature support, no adequate attempt)**

**The FLAGs drove real self-correction** — the strategizer revised H1 and H2 in
response (advise-with-teeth doing its job). And #9 *agreed with the gate critic*:
both independently flagged H1's §4 problem first — the shared-charter design (Q2=i)
holds.

**But three real limitations surfaced — friction first:**
1. **Whipsaw.** On H1, #9 flagged FALSIFIED (§4), then flagged the INCONCLUSIVE
   retry (§3, "should be FALSIFIED"), *pushing the agent back toward FALSIFIED* —
   which the gate critic then **REJECTED** as a §4 goalpost-move. #9's §3 push
   actively steered toward the verdict the gate rejected.
2. **Non-determinism.** #9 gave **opposite verdicts on the same H1 INCONCLUSIVE
   status** — flagged it §3 once, validated it "no charter concern" later. Same
   status, same hypothesis, different ruling. LLM-judge variance.
3. **Local view.** #9 judges each `HypothesisUpdate` in isolation (hypothesis +
   proposed status + cited delegation). It missed the *cross-update*
   interpretation-switch (the goalpost-move spanning FALSIFIED→reinterpret) that
   the critic caught by reading the full status_log + delegation history.

**Verdict.** #9 is a valuable early, charter-grounded nudge that drives engagement
and self-correction — but it is fallible (non-deterministic, locally scoped,
whipsaw-prone) and must not be treated as authoritative; the gate critic remains
the source of truth. This **vindicates Q1=(B) advise-non-blocking**: had #9
blocked, its whipsaw/non-determinism would have wedged the run. The kill switch
(`F3DASM_VERDICT_VALIDATOR`) is the right safety valve.

**Design idea (for the user — §4):** give #9 the full status_log + delegation
history for the hypothesis, not just the single update, so it stops whipsawing.
The local view is the root of limitations 1–3.

## Time starvation (#10 + slow literature review) — THE binding constraint

The dominant failure mode across the audit. Both audit runs hit the 60-min watchdog.

- **Run #1:** ~25 min literature review + a long single implementer campaign →
  never reached the deliverable phase. #9/#13 went **unexercised**.
- **Run #2:** **skipped** the literature review → reached the deliverable, got the
  critic to **PASS** on attempt 3 — but the 3-attempt gate loop (REJECT on the §4
  issue #9 surfaced → REVISE on a doc conflict → PASS) pushed it past 60 min.
- **Baseline run #3 GATED in 26:41** — also skipped lit, single clean gate.

So the 60-min budget is too tight for *lit review (~25 min) + campaign + notebook +
multi-attempt gate*. Levers: cap/optionalize the literature review; #10 (monolithic
campaign) still binds; the late doc-conflict REVISE (run_summary.md vs ledger) is a
recurring end-stage time sink worth a notebook↔ledger sync helper; or revisit the
60-min watchdog vs the work required.

## Recommendations (prioritized)

1. **Fix #14 reap** (HIGH) — recursive PPID tree-walk; stops a CPU-core leak on
   every watchdog kill.
2. **Attack time starvation** — cap or default-off the literature review (~25-min
   sink); revisit #10 (monolithic campaign) and/or the 60-min watchdog. This is
   what actually blocks GATED outcomes, not #9/#13.
3. **#9: keep, don't promote** — valuable nudge, keep kill-switchable, do NOT
   escalate toward blocking. Consider feeding it the full hypothesis history to
   cure the whipsaw/non-determinism (§4 decision).
4. **Notebook↔ledger doc-sync** — the run_summary.md-vs-ledger REVISE burned run
   #2's watchdog margin at the wire; derive run_summary from the ledger or add a
   sync check.
5. **Literature retrospective gap** — make the lit reviewer write a retro like
   every other node.
